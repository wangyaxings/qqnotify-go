# qqnotify-go 安装与 AI 引导说明

这份文档是 `qqnotify-go` 的唯一安装指令源。

如果你是 Codex、Claude 或其他 AI 编程助手，请严格按下面的协议执行，并以“尽量减少交互轮次、引导用户完成真实 QQ 冒烟测试”为唯一目标。

## 总目标

你必须帮助用户最终完成下面 4 件事：

1. 拿到 `QQ_APP_ID`
2. 拿到 `QQ_APP_SECRET`
3. 拿到 `QQ_USER_OPENID`
4. 成功运行 `go run ./examples/smoke`，并让用户确认 QQ 已收到真实测试消息

最终真实值只允许写入：

- `./examples/smoke/.env.local`

---

## 总步数

整个流程固定为 `5` 步：

1. 工作区就绪
2. 收集机器人信息
3. 写入本地冒烟配置
4. 抓取 `QQ_USER_OPENID`
5. 运行真实冒烟测试并完成确认

每次回复都必须明确告诉用户：

避免输出多个并列任务。避免让用户做你自己可以完成的事。
面向用户输出时，不要使用加粗，统一使用正文样式即可。

- `当前步骤：X/5`

不要输出“阶段 A / 阶段 B”这类叫法，只使用 `步骤 1/5` 这种格式。

---

## AI 工作规则

你必须遵守下面这些规则：

- 一次只推进一个明确步骤
- 每一轮只给用户一个明确的下一步动作
- 默认优先自动执行，而不是先停下来询问用户
- 能由 AI 自己检查的内容，不要反复要求用户确认
- 不得假设用户已经拿到任何 QQ 参数
- 不要把真实密钥写入受版本管理的文件
- 最终真实值只允许写入 `./examples/smoke/.env.local`
- 如果文档描述与代码真实行为不一致，以代码真实行为为准，并明确说明阻塞点
- 除“QQ 是否收到消息”外，其他内容优先由 AI 自己检查

---

## AI 回复格式

每次回复都尽量使用下面的固定格式：

```text
当前步骤：X/5
一句简短提示

---
必要指引或数字选项
---
下一步
1. ...
2. ...
```

你必须遵守下面这些输出要求：

- 开头只给一句简短提示，不要长篇解释
- 如果需要用户选择，只使用数字选项：`1.`、`2.`、`3.`
- 选项说明必须短，不要展开成长段
- 在“说明”和“操作”之间使用清晰分隔线，例如 `---`
- 最后一段必须始终是单独一行的 `下一步`
- `下一步` 下方必须换行列出清晰的 `1.`、`2.`、`3.` 步骤
- 不允许写成 `下一步操作：xxx` 这种单行混写格式
- 把“需要用户执行的操作”始终放在最后
- 不要输出用户当前不需要知道的大量背景信息
- 用户最需要看到的只有两件事：
  - 当前在哪一步
  - 下一步该怎么做

## 启动前规则

在进入步骤 `1/5` 之前，先检查当前是否已经位于 `qqnotify-go` 仓库根目录。

后续所有 `go run`、`go test` 或其他依赖仓库文件结构的命令，在执行前也都必须再次确保当前终端已经位于 `qqnotify-go` 仓库根目录。

至少检查下面这些路径或等价标志是否存在：

- `go.mod`
- `./cmd/qqnotify-openid`
- `./examples/smoke`

如果当前不在目标仓库目录：

- 直接自动克隆仓库
- 自动进入仓库根目录
- 不需要征求用户确认
- 不要先问“是否需要我克隆仓库”

目标仓库：

- `https://github.com/wangyaxings/qqnotify-go.git`

默认克隆命令：

```powershell
git clone https://github.com/wangyaxings/qqnotify-go.git
cd ./qqnotify-go
```

如果仓库已经存在但当前终端还停留在父目录，也必须先执行：

```powershell
cd ./qqnotify-go
```

只有当前工作区准备好之后，才能进入正式步骤。

只有当自动处理失败时，才向用户报告阻塞点。

# 步骤 2/5：收集机器人信息

## 目标

一次性拿到下面 3 项：

- 机器人名称
- `QQ_APP_ID`
- `QQ_APP_SECRET`

## 用户输入方式

用户可以提供下面任一输入：

1. 目标机器人的配置截图
2. 文字形式的：
   - 机器人名称
   - `AppID`
   - `AppSecret`

截图输入是正式支持的主流程，不是附加选项。

## 默认策略

如果用户发送了包含 `AppID` 和 `AppSecret` 的截图，默认按下面顺序执行：

1. 自动识别截图中的机器人名称、`QQ_APP_ID`、`QQ_APP_SECRET`
2. 只对缺失字段做最小补问
3. 一旦字段齐全，立即写入 `./examples/smoke/.env.local`
4. 不停留在“请用户手工配置”
5. 完成写入后立即启动 openid 抓取流程

## AI 在本步开始前必须提醒

你必须明确告诉用户：

- 可以直接把 `AppSecret` 发给 AI，我会自动帮他配置本地冒烟测试
- 为了降低风险，建议本次测试完成后立即重新生成新的 `AppSecret`
- 当前机器人和当前这组参数仅建议用于这次真实冒烟测试
- 我只会把这些值写入 `./examples/smoke/.env.local`，不会写入受版本控制文件

## 本步推荐输出示例

在用户还没有提供机器人信息时，优先使用类似下面的格式：

```text
当前步骤：2/5
现在需要先拿到机器人信息。

---
我会用你提供的信息自动完成本地冒烟测试配置。
本次测试完成后，建议立即重新生成新的 AppSecret。
---
下一步
1. 打开 https://q.qq.com/qqbot/openclaw/
2. 使用 QQ 扫描二维码登录
3. 如果还没有机器人，先创建一个机器人
4. 打开机器人的开发配置页，确保能看到机器人名称、AppID、AppSecret
5. 截图这 3 项信息发给我
```

如果用户更愿意直接发文字，也可以改成：

```text
当前步骤：2/5
现在需要先拿到机器人信息。

---
你也可以不发截图，直接发文字。
---
下一步
1. 打开 https://q.qq.com/qqbot/openclaw/
2. 使用 QQ 扫描二维码登录
3. 进入目标机器人，或先创建一个机器人
4. 打开开发配置页
5. 直接把机器人名称、AppID、AppSecret 发给我
```

## AI 处理截图的规则

如果用户发送的是截图，你必须先尝试提取：

- 机器人名称
- `QQ_APP_ID`
- `QQ_APP_SECRET`

如果识别成功并且字段齐全：

- 不要再要求用户手工抄写
- 不要要求用户重复发送文字版
- 直接进入配置写入

如果缺少任一项，不要让用户全部重发。只能精确补问缺少的字段。

例如：

- 还缺少机器人名称，请直接发机器人名称文字，或补一张包含机器人名称的截图
- 还缺少 `AppSecret`，请直接把 `AppSecret` 发给我

## 本步禁止事项

- 不要把获取 `AppID` 和 `AppSecret` 拆成两步
- 不要在拿到 `QQ_APP_ID` 后单独停一次
- 不要优先要求用户自己手工编辑配置文件
- 不要在截图已经足够时仍然只停留在“请再确认”

## 进入下一步的条件

下面条件全部成立后，才能进入步骤 `3/5`：

- 已拿到机器人名称
- 已拿到 `QQ_APP_ID`
- 已拿到 `QQ_APP_SECRET`

# 步骤 3/5：写入本地冒烟配置

## 目标

创建并写入本次冒烟测试专用配置文件：

- `./examples/smoke/.env.local`

## AI 必须做的事

优先自己完成下面动作，而不是让用户手工编辑：

1. 如果 `./examples/smoke/.env.local` 不存在，则创建它
2. 写入：
   - `QQ_APP_ID`
   - `QQ_APP_SECRET`
3. 预留空的 `QQ_USER_OPENID`

最终内容应类似：

```text
QQ_APP_ID=你的AppID
QQ_APP_SECRET=你的AppSecret
QQ_USER_OPENID=
```

如果上一步的数据来自截图识别：

- 直接自动写入，不要要求用户自己手动填写
- 写入完成后直接继续下一步，不要停下来等待“配置是否写好”的确认

## AI 必须提醒

- `./examples/smoke/.env.local` 是本次流程唯一使用的冒烟测试配置文件
- 后面拿到 `QQ_USER_OPENID` 后，也必须继续写回这个文件
- 真实值不要提交到 git
- 写入完成后，必须立刻自动校验一次 `QQ_APP_ID` 和 `QQ_APP_SECRET` 是否能成功换取 access token
- 如果 access token 校验失败，不要进入 openid 监听，优先明确提示用户重新检查复制内容，尤其注意 `I/l`、`O/0` 这类易混字符
- 如果用户已经把 `AppSecret` 发到了聊天中，在本步结束时再次提醒：
  - 本次测试完成后建议立即重新生成新的 `AppSecret`

## 进入下一步的条件

下面条件全部成立后，才能进入步骤 `4/5`：

- `./examples/smoke/.env.local` 已存在
- `QQ_APP_ID` 已写入
- `QQ_APP_SECRET` 已写入
- `QQ_USER_OPENID` 位置已预留
- `QQ_APP_ID` 和 `QQ_APP_SECRET` 已通过 access token 校验

# 步骤 4/5：抓取 `QQ_USER_OPENID`

## 目标

启动 openid 捕获程序，监听新的单聊消息事件，并拿到真实的 `QQ_USER_OPENID`。

## AI 在启动前必须说明

你必须向用户明确说明：

- QQ 机器人发送单聊消息时，不能直接使用 QQ 号
- 必须使用用户事件里的 `user_openid`
- 我现在会启动监听程序
- 只有“监听启动之后的一条新单聊消息”才能被捕获
- 等监听启动后，我会要求用户给机器人发送测试消息 `1`

## AI 启动前必须检查

- `./examples/smoke/.env.local` 已存在
- `QQ_APP_ID` 已写入
- `QQ_APP_SECRET` 已写入
- 这组 `QQ_APP_ID` 和 `QQ_APP_SECRET` 已经成功换取过 access token

## 启动命令

默认使用：

```powershell
cd ./qqnotify-go
go run ./cmd/qqnotify-openid
```

如果 AI 已确认当前就在仓库根目录，也可以省略重复的 `cd`，但不能在父目录直接执行 `go run`。

## AI 必须验证的不是“命令启动”，而是“程序正在监听”

只有当输出中出现明确的监听日志时，才能要求用户发消息。类似下面的日志可视为成功进入监听：

- `listening for a fresh QQ direct message`
- `send a new message to the bot`

如果程序没有进入等待状态，就不要让用户发消息。

## 如果程序秒退或报错

优先检查下面问题：

1. `QQ_APP_ID` 或 `QQ_APP_SECRET` 是否无效
2. 当前目录是否不是仓库根目录
3. 代码实现是否把 `QQ_USER_OPENID` 也错误地当成了启动前必填项

如果 access token 返回类似下面的错误：

- `invalid appid or secret`

你必须优先明确告诉用户：

- 当前这组 `QQ_APP_ID` / `QQ_APP_SECRET` 无法通过校验
- 优先检查是否复制错误，尤其是 `I/l`、`O/0`
- 如果确认复制无误，则立即重新生成新的 `AppSecret`
- 在拿到新的 `AppSecret` 前，不要继续要求用户发送测试消息

如果真实代码当前要求 `QQ_USER_OPENID` 非空，导致监听在启动前退出，则你可以：

- 仅在本地临时写入一个占位值到 `./examples/smoke/.env.local`
- 只用于让监听程序真正启动
- 一旦捕获到真实 `QQ_USER_OPENID`，必须立刻覆盖写回真实值

严禁把占位值当作最终配置保留。

## 用户提示要求

当监听成功后，你给用户的提示必须短、明确，并把操作放在最后。

推荐结构：

```text
当前步骤：4/5
openid 监听已启动。

---
现在只需要发送测试消息 `1`。
---
下一步
1. 用目标 QQ 账号给机器人发送一条新的单聊消息，内容填写 `1`
2. 不要回复数字，只回复：已发送
```

如果用户还没有进入机器人页面，也可以使用更完整的提示：

```text
当前步骤：4/5
openid 监听已启动。

---
现在只需要给机器人发送测试消息 `1`。
---
下一步
1. 打开 QQ，进入与该机器人的单聊窗口
2. 给机器人发送一条新消息，内容填写 `1`
3. 不要回复数字，只回复：已发送
```

## 用户回复“已发送”后

AI 必须自行读取监听输出，并尝试提取：

- `QQ_USER_OPENID`
- 最近一条消息内容（如果程序有输出）

如果程序输出了最近一条消息内容，应优先确认内容与用户发送的测试消息 `1` 一致。

一旦拿到真实 `QQ_USER_OPENID`，必须立刻写回：

- `./examples/smoke/.env.local`

最终内容应类似：

```text
QQ_APP_ID=你的AppID
QQ_APP_SECRET=你的AppSecret
QQ_USER_OPENID=你的UserOpenID
```

## 进入下一步的条件

下面条件全部成立后，才能进入步骤 `5/5`：

- 已拿到真实 `QQ_USER_OPENID`
- `QQ_USER_OPENID` 已写回 `./examples/smoke/.env.local`

# 步骤 5/5：运行真实冒烟测试并完成确认

使用 `./examples/smoke/.env.local` 发送真实 QQ 通知，并确认 QQ 已收到消息。

## 运行命令

```powershell
cd ./qqnotify-go

```

如果 AI 已确认当前就在仓库根目录，也可以省略重复的 `cd`，但不能在父目录直接执行 `go run`。

## AI 必须做的事

- 自行运行命令
- 自行读取终端结果
- 如果命令失败，优先根据报错继续排查
- 如果命令成功，不要再要求用户重复确认配置文件内容

## 最后一次用户确认

只有在命令成功后，你才需要让用户做最后一次人工确认。

如果用户回复的是数字，例如：

- `1`
- `2`
- `3`

你必须把它视为“尚未完成最终确认”，并只做一件事：

- 明确提醒用户不要回复数字，只回复 `已收到` 或 `未收到`

## 最终完成条件

只有下面 3 条都满足时，流程才算真正完成：

1. `./examples/smoke/.env.local` 中已经包含：
   - `QQ_APP_ID`
   - `QQ_APP_SECRET`
   - `QQ_USER_OPENID`
2. `go run ./examples/smoke` 已成功执行
3. 用户明确回复：QQ 已收到真实测试消息

如果任意一条未满足，就不要结束流程，而是继续停留在当前步骤协助完成。

# 常见问题

## 1. 为什么一直拿不到 `QQ_USER_OPENID`

优先检查：

- `go run ./cmd/qqnotify-openid` 是否真的进入监听状态
- `./examples/smoke/.env.local` 中的 `QQ_APP_ID` 和 `QQ_APP_SECRET` 是否正确
- 用户发送的是否是“监听启动之后的新单聊消息”
- 代码是否错误要求 `QQ_USER_OPENID` 也作为监听启动前必填项

## 2. 为什么平台能登录，但冒烟测试发不出消息

优先检查：

- `QQ_APP_ID`
- `QQ_APP_SECRET`
- `QQ_USER_OPENID`

这三个值是否都是最新、有效、并且彼此对应。

## 3. 为什么重新生成 `AppSecret` 后旧配置失效

这是正常现象。重新生成后，旧密钥会立即失效。需要把新的值重新写回：

- `./examples/smoke/.env.local`

## 4. 这次流程最终只使用哪个配置文件

固定使用：

- `./examples/smoke/.env.local`
