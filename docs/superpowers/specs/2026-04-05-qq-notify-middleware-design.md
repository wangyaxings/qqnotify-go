# Go QQ 通知中间件仓库定位设计

**日期：** 2026-04-05  
**状态：** 已确认方向，待进入实施计划

## 1. 背景

当前项目已经具备基于 QQ 官方机器人链路发送消息和接收消息的基础能力，但代码组织仍然偏向单项目实验形态，产品定位也更接近“当前项目可用”，还没有收敛成一个面向更广泛 Go 开发者的独立中间件仓库。

新的目标不是继续演进成一个“全功能 QQ Bot 框架”，而是抽出其中最有传播力、最容易被采用的一条能力线，做成一个真正适合开源增长的 Go 仓库。

## 2. 产品定位

本仓库的核心定位如下：

- 一个面向 AI 服务与自动化任务的 QQ 通知中间件
- 帮助 Codex、AI Agent、脚本任务、CI/CD、定时任务与内部工具，以最少配置把结果稳定发送到 QQ
- 第一阶段只聚焦“发通知”，不扩展成全功能机器人框架

英文一句话定位建议：

> A production-ready Go middleware for sending AI and automation notifications to QQ.

更强调传播的版本：

> The fastest way for Go apps, agents, and Codex workflows to send notifications to QQ.

## 3. 目标用户

首批目标用户限定为以下三类：

- 想把 Codex 或 AI Agent 的执行结果发到 QQ 的开发者
- 想把脚本任务、定时任务、CI/CD 结果发到 QQ 的工程师
- 想给内部 AI 工具或自动化平台增加 QQ 通知出口的团队

这些用户关注的不是“Bot 框架功能是否齐全”，而是：

- 是否容易接入
- 是否稳定
- 是否适合生产使用
- 是否足够贴近 AI / 自动化场景

## 4. 非目标

为了保持仓库聚焦，第一阶段明确不做以下内容：

- 不做完整聊天机器人框架
- 不做双向消息系统
- 不做命令分发框架
- 不做会话管理
- 不做插件系统
- 不做群管理平台
- 不优先抽象成多平台统一通知框架

这些能力可以作为后续可能演进方向，但不能影响 v1 的通知中间件定位。

## 5. 为什么这个定位更有机会获得 300 Star

如果仓库只是“又一个 Go QQ Bot SDK 封装”，会直接落入与官方 SDK 及现有轻量封装相似的区间，差异化不足，传播语义也不够强。

相比之下，“让 AI 和自动化任务快速把消息发到 QQ”具备更清晰的增长优势：

- 场景具体，别人更容易一句话复述仓库价值
- 可服务的人群不只限于 QQ Bot 开发者，还能覆盖 AI 工具开发者、自动化工程师、独立开发者
- README、示例、文章、演示都可以围绕高价值使用场景展开
- 用户购买点不是 API 完整度，而是接入速度、稳定性和场景适配度

因此，这个仓库的竞争对象不应定义为“所有 QQ SDK”，而应定义为：

- 手写一段通知逻辑
- 临时脚本拼接调用
- 不稳定、不可复用的内部实现

仓库必须提供一个明显更强的替代方案：更快、更稳、更省心。

## 6. v1 核心产品承诺

v1 只承诺 4 件事：

### 6.1 五分钟接入

用户填入最少的配置后，就能发出第一条 QQ 消息。

### 6.2 为 AI / 自动化场景而生

仓库首页、示例、文案全部直接面向以下场景：

- Codex
- AI Agents
- CI/CD
- Cron Jobs
- Internal Tools
- Alerts / Reports

### 6.3 生产可用

不仅能发消息，还必须具备线上可用的基础能力：

- 超时控制
- 重试策略
- 错误分类
- 基本日志能力
- 配置校验

### 6.4 默认体验极简

首次使用必须足够轻：

- 一个最小 Go API
- 一个最小 Quick Start
- 一个可直接运行的示例

## 7. v1 必做能力

### 7.1 极简 Go API

用户应能通过非常直接的接口发通知，例如：

```go
client.SendText(ctx, "Codex task finished successfully")
```

或者通过结构化通知对象发送：

```go
client.Send(ctx, Notification{
    Title:   "任务完成",
    Body:    "Codex 已完成代码修改并通过测试",
    Status:  "success",
    Source:  "codex",
    TraceID: "job-123",
})
```

### 7.2 稳定的通知发送层

需要包含：

- 文本发送能力
- 统一配置加载与校验
- 请求超时
- 重试机制
- 错误返回清晰化
- 基本日志接口或钩子

### 7.3 AI / 自动化友好的消息模型

即使最终发出去的是文本，也应支持结构化输入，便于：

- Codex
- Agent
- CI
- 定时任务
- 监控告警

共用同一套中间件接口。

### 7.4 HTTP Bridge

提供一个轻量 HTTP 接口，允许非 Go 系统也通过 HTTP 调用该仓库完成 QQ 通知发送。

这将显著扩大适用范围，使仓库不仅仅服务于 Go 项目本身，也能服务于任意支持 HTTP 的自动化系统。

### 7.5 强示例体系

至少提供以下示例：

- `examples/codex`
- `examples/cron`
- `examples/github-actions`
- `examples/http-bridge`

## 8. v1 不做能力

为了保证第一阶段聚焦，以下内容明确延后：

- 双向消息接收
- 命令路由
- 插件机制
- 会话状态管理
- 群聊复杂能力
- 多平台统一抽象
- 复杂消息编排 DSL

## 9. 仓库命名策略

命名应优先体现“QQ 通知”而不是“Bot Framework”。

推荐命名方向：

- `qqnotify-go`
- `go-qq-notify`
- `qqnotify`

首选建议：

> `qqnotify-go`

命名策略原则：

- 仓库名聚焦“QQ 通知”
- README 副标题强调“AI / automation”
- 不将名称过度绑定到 `codex`，避免限制未来场景扩展

## 10. README 首屏设计

README 首屏只做 5 件事：

### 10.1 一句话价值主张

让用户在 10 秒内知道仓库是干什么的。

### 10.2 15 秒示例代码

展示最小发送流程，降低第一次理解成本。

### 10.3 适用场景列表

直接列出：

- Codex / AI Agents
- CI/CD
- Cron Jobs
- Internal Tools
- Alerts and Reports

### 10.4 为什么不用自己手写

强调：

- retries
- timeouts
- structured payload
- clean errors
- HTTP bridge
- examples

### 10.5 Quick Start

给出最短路径：

- 安装
- 环境变量配置
- 发送第一条 QQ 通知

README 首页不应堆积：

- 长篇设计说明
- 大量 API 列表
- 复杂架构图
- 大而全路线图

## 11. 增长策略

### 11.1 增长主线

本仓库应采用“场景驱动增长”，而非“技术抽象驱动增长”。

核心传播语句应围绕高价值使用场景展开，例如：

- 让 Codex 执行完成后自动发 QQ
- 让 GitHub Actions 失败时发 QQ
- 给定时任务生成的日报发 QQ
- 给内部 AI 工具增加 QQ 通知出口

### 11.2 为什么不主打“又一个 SDK”

“QQ Bot SDK”描述过于宽泛，不利于增长。  
“AI / 自动化的 QQ 通知中间件”更具体、更好理解，也更容易形成转发与推荐。

## 12. 路线图

### 12.1 前 30 天目标

目标是发布一个“可理解、可跑通、可传播”的 v1。

需要完成：

- 收紧仓库定位、命名与文案
- 发布稳定的通知发送核心
- 补 3 到 5 个高价值示例
- 打磨 README
- 建立基础工程规范：测试、CI、版本、License、变更说明

成功标准：

- 用户 5 分钟能跑通
- 用户 1 分钟知道适不适合自己
- 用户愿意转发或 star

### 12.2 前 90 天目标

目标是让仓库开始被引用、被嵌入、被讨论。

需要完成：

- HTTP bridge 稳定化
- 补齐面向 Codex、CI、cron、内部工具的接入示例
- 输出多篇围绕真实场景的文章或演示内容
- 收集首批用户反馈并优化 API 与文档

成功标准：

- 有真实用户开始接入
- 有 issue、PR、反馈和二次传播
- 仓库开始被视作一个现成解决方案

### 12.3 半年目标

目标是在不丢失定位的前提下，增强产品厚度。

可考虑加入：

- 模板化消息
- 更强的错误分类
- 可观测性接口
- 企业网络环境友好支持
- transport / webhook 可扩展层

但仍不应过早转向全功能 Bot Framework。

## 13. 成功指标

第一阶段的成功不以功能数量衡量，而以传播与采用衡量。

建议关注：

- GitHub stars 增长趋势
- README 到示例的转化效果
- 真实使用场景数量
- issue / discussion / PR 活跃度
- 外部文章和引用次数

## 14. 结论

这个仓库要想真正接近 300 stars，最开始就必须避免“做大做全”的诱惑，而是把自己定义成：

> 一个面向 AI 服务与自动化任务的、最快接入、生产可用的 QQ 通知中间件。

只有先把这个单点价值做得足够锋利，后续才有资格从通知中间件逐步扩展到更大的 QQ 机器人能力集合。
